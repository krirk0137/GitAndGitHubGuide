01

Git, GitHub และ GitHub Desktop ต่างกันยังไง

เคลียร์ความสับสนที่หลายคนเข้าใจผิดตั้งแต่จุดเริ่มต้น

สามคำนี้คนละสิ่งกัน แต่ทำงานร่วมกัน วิธีจำง่ายที่สุด: Git คือ "เครื่องมือ", GitHub คือ "บ้านบนคลาวด์", GitHub Desktop คือ "รีโมตคอนโทรล"

⚙️Git

โปรแกรม Version Control ที่ติดตั้งในเครื่องเรา ทำหน้าที่บันทึกประวัติการเปลี่ยนแปลงโค้ดทุกเวอร์ชัน ทำงานผ่าน คำสั่ง (command line) ใช้งานได้แม้ไม่มีอินเทอร์เน็ต — Git คือพระเอกตัวจริง

☁️GitHub

บริการเว็บสำหรับเก็บโปรเจกต์ Git ไว้บนคลาวด์ เพื่อให้ทีมแชร์โค้ดกันได้ มีระบบ Pull Request, Issues, Code Review, CI/CD — คู่แข่งคล้ายกันคือ GitLab, Bitbucket, Azure DevOps

🖱️GitHub Desktop

โปรแกรม หน้าจอกราฟิก (GUI) ที่ครอบ Git ไว้ ให้เรากดปุ่มแทนพิมพ์คำสั่ง เหมาะกับคนเริ่มต้น เห็น diff ชัด ลดโอกาสพิมพ์คำสั่งผิด — แต่เบื้องหลังก็ยังเรียก Git ตัวเดิม

💡

เปรียบเทียบให้เห็นภาพ: Git เหมือน "เครื่องยนต์รถ", GitHub เหมือน "อู่/ลานจอดรถส่วนกลางที่ทีมเอารถมาจอดรวมกัน", ส่วน GitHub Desktop คือ "พวงมาลัยกับแผงควบคุม" ที่ทำให้ขับง่ายขึ้น คุณไม่จำเป็นต้องถอดเครื่องยนต์เป็นถึงจะขับรถได้ แต่รู้ไว้บ้างก็ดีตอนรถมีปัญหา

🎯ในที่ทำงานจริง คุณจะเจออะไร

  • โค้ดของบริษัทถูกเก็บไว้บน GitHub (หรือ GitLab/Azure DevOps/Bitbucket — แนวคิดเหมือนกันหมด)
  • คุณ clone โปรเจกต์ลงเครื่อง → แก้โค้ดบน branch ของตัวเอง → push ขึ้นไป → เปิด Pull Request ให้เพื่อนรีวิว → merge เข้า branch หลัก
  • จะใช้ผ่าน command line หรือ GitHub Desktop ก็ได้ หลายคนใช้ทั้งสองอย่างผสมกัน (GUI สำหรับงานทั่วไป, command line ตอนต้องทำอะไรซับซ้อน)
02

โมเดลหลัก: 3 พื้นที่ของ Git

ถ้าเข้าใจภาพนี้ภาพเดียว คำสั่ง Git ที่เหลือจะเข้าใจตามมาทั้งหมด

หัวใจของ Git คือไฟล์เดินทางผ่าน 4 พื้นที่ ไฟล์จะค่อยๆ ขยับจากซ้ายไปขวาด้วยคำสั่งต่างๆ ลองดูแอนิเมชันด้านล่าง (จุดสว่างคือไฟล์ที่กำลังเดินทาง)

git add git commit git push git pull / fetch
พื้นที่คืออะไรคำสั่งที่เกี่ยวข้อง
Working Directoryโฟลเดอร์โปรเจกต์จริงที่เราเปิดแก้ไฟล์อยู่ตอนนี้git status ดูว่าไฟล์ไหนเปลี่ยน
Staging Area
(Index)		"ตะกร้า" ที่เลือกว่าจะเอาไฟล์ไหนเข้า commit รอบนี้บ้างgit add หยิบไฟล์เข้าตะกร้า
Local Repositoryประวัติ commit ทั้งหมดที่บันทึกไว้ในเครื่อง (โฟลเดอร์ .git)git commit บันทึกถาวร
Remoteสำเนาบนเซิร์ฟเวอร์ เช่น GitHub ที่ทุกคนในทีมเข้าถึงได้git push / git pull
✅

จำประโยคนี้ไว้: "แก้ไฟล์ → add เข้าตะกร้า → commit เซฟลงประวัติ → push ส่งขึ้นคลาวด์" วนแบบนี้ทุกวันคือ 90% ของงาน Git ในชีวิตประจำวัน

03

ติดตั้งและตั้งค่าครั้งแรก

ทำครั้งเดียวต่อเครื่องใหม่ — วันแรกที่ได้โน้ตบุ๊กบริษัทมาก็ทำตามนี้ได้เลย

📥1. ติดตั้งและเช็กเวอร์ชัน

ดาวน์โหลด Git จาก git-scm.com (Windows/Mac/Linux) จากนั้นตรวจสอบว่าติดตั้งสำเร็จ:

terminal
git --version
# ตัวอย่างผลลัพธ์: git version 2.45.x

🪪2. ตั้งชื่อและอีเมล (สำคัญมาก)

ทุก commit จะถูกเซ็นชื่อด้วยข้อมูลนี้ ใช้อีเมลเดียวกับที่สมัคร GitHub/บัญชีบริษัท เพื่อให้ commit ผูกกับโปรไฟล์ถูกต้อง:

ตั้งค่าระดับ global (ทั้งเครื่อง)
git config --global user.name "Krirk Dev"
git config --global user.email "you@company.com"

# ตั้งให้ branch แรกชื่อ main เสมอ (มาตรฐานปัจจุบัน)
git config --global init.defaultBranch main

# ให้ VS Code เป็น editor เวลา git ต้องการให้พิมพ์ข้อความ
git config --global core.editor "code --wait"
⚠️

เครื่องบริษัทบางที่อาจต้องใช้อีเมลคนละตัวกับงานส่วนตัว ถ้าทำหลายโปรเจกต์ ตั้งค่าเฉพาะโปรเจกต์ได้โดยตัด --global ออก (รันในโฟลเดอร์นั้น) เพื่อ override เฉพาะที่

🔑3. ตั้งค่าการยืนยันตัวตน (Authentication)

ตอน push/pull กับ GitHub ต้องพิสูจน์ว่าเป็นเรา มี 2 วิธีหลัก เลือกอย่างใดอย่างหนึ่ง:

🔐SSH Key (แนะนำสำหรับ command line)

สร้างกุญแจครั้งเดียว แล้วไม่ต้องกรอกรหัสอีก เหมาะกับคนใช้ terminal บ่อย

🎟️HTTPS + Token

ใช้ URL แบบ https แล้วยืนยันด้วย Personal Access Token (PAT) แทนรหัสผ่าน (GitHub เลิกรับรหัสผ่านธรรมดาแล้ว) — GitHub Desktop จัดการให้อัตโนมัติ

สร้าง SSH key
# 1) สร้างกุญแจ (กด Enter ผ่านได้หมด)
ssh-keygen -t ed25519 -C "you@company.com"

# 2) ก๊อปปี้ public key ไปวางใน GitHub → Settings → SSH keys
# Windows (Git Bash):
cat ~/.ssh/id_ed25519.pub

# 3) ทดสอบการเชื่อมต่อ
ssh -T git@github.com
💡

ถ้าใช้ GitHub Desktop เรื่อง auth นี้ไม่ต้องคิดมาก — แค่ Sign in ด้วยบัญชี GitHub ในแอป มันจัดการ token ให้เองทั้งหมด นี่คือข้อดีใหญ่ของ GUI

🚫4. ตั้ง .gitignore (อย่าลืมเด็ดขาด)

ไฟล์ .gitignore บอก Git ว่าไฟล์ไหน "ไม่ต้องเก็บ" เช่น ไฟล์ build, secret, โฟลเดอร์ขยะ — สำหรับโปรเจกต์ .NET ตัวอย่างมาตรฐาน:

.gitignore (ตัวอย่าง .NET / Visual Studio)
# build outputs
bin/
obj/
# Visual Studio
.vs/
*.user
*.suo
# ความลับ — ห้าม commit เด็ดขาด!
appsettings.Development.json
*.env
secrets.json
# IDE / OS อื่นๆ
.idea/
.DS_Store
🛑

กฎเหล็ก: อย่า commit รหัสผ่าน, API key, connection string จริง ขึ้น Git เด็ดขาด แม้จะลบทีหลังมันก็ยังอยู่ในประวัติ คนเข้าถึง repo ได้จะเห็นหมด — นี่คือความผิดพลาดที่ทำให้คนโดนไล่ออกมาแล้ว ใช้ User Secrets / Environment Variables / Key Vault แทน

04

Workflow ประจำวัน

วงจรคำสั่งที่คุณจะพิมพ์ซ้ำๆ ทุกวันจนติดมือ

📂เริ่มต้นกับโปรเจกต์

terminal
# ดึงโปรเจกต์ที่มีอยู่แล้วจาก GitHub ลงเครื่อง (กรณีที่พบบ่อยสุดในที่ทำงาน)
git clone git@github.com:company/project.git

# หรือเริ่มโปรเจกต์ใหม่จากศูนย์
git init

🔄วงจรหลัก: status → add → commit → push

terminal
# 1) ดูว่ามีไฟล์อะไรเปลี่ยนบ้าง (ใช้บ่อยที่สุด — พิมพ์บ่อยๆ ได้เลย)
git status

# 2) หยิบไฟล์เข้า staging
git add index.html          # เฉพาะไฟล์เดียว
git add .                   # ทุกไฟล์ที่เปลี่ยน
git add -p                  # เลือกทีละส่วน (interactive)

# 3) บันทึก commit พร้อมข้อความอธิบาย
git commit -m "feat: add login validation"

# 4) ส่งขึ้น remote
git push

⬇️ดึงงานใหม่ของทีมลงมา

terminal
# pull = fetch (ดึงข้อมูล) + merge (รวมเข้า) ในคำสั่งเดียว
git pull

# fetch อย่างเดียว = ดึงมาดูก่อน ยังไม่รวม (ปลอดภัยกว่า)
git fetch
git log origin/main --oneline   # ดูว่ามีอะไรใหม่ก่อนตัดสินใจ merge
✅

นิสัยที่ดี: pull ก่อน push เสมอ ตอนเริ่มงานแต่ละวัน ดึงงานล่าสุดของทีมลงมาก่อน จะได้ไม่ชนกับงานคนอื่นและลด conflict

🔍ดูประวัติและความเปลี่ยนแปลง

terminal
git log --oneline --graph --all   # ดูประวัติแบบกราฟสวยๆ
git diff                          # ดูสิ่งที่แก้แต่ยังไม่ add
git diff --staged                 # ดูสิ่งที่ add แล้วรอ commit
git show <commit-hash>            # ดูรายละเอียด commit หนึ่งๆ
05

Branch & Merge — หัวใจของการทำงานเป็นทีม

แตกสายงานออกไปทำของตัวเอง โดยไม่กระทบงานหลัก แล้วค่อยรวมกลับ

Branch คือ "เส้นเวลาคู่ขนาน" คุณก๊อปสภาพโค้ดปัจจุบันออกมาเป็นสายของตัวเอง ทดลองแก้ได้เต็มที่ ถ้าพังก็ไม่กระทบ main เมื่อเสร็จและผ่านรีวิวแล้วจึง merge รวมกลับเข้า main

main (branch หลัก) feature branch (ของเรา)
การทำงานกับ branch
# ดู branch ทั้งหมด (ดอกจันคือ branch ที่อยู่ตอนนี้)
git branch

# สร้าง branch ใหม่ + ย้ายไปอยู่เลย (switch คือคำสั่งใหม่ที่ชัดกว่า checkout)
git switch -c feature/login
# เทียบเท่าคำสั่งเก่า: git checkout -b feature/login

# ทำงาน... add... commit... แล้ว push branch ขึ้น remote ครั้งแรก
git push -u origin feature/login

# สลับกลับไป main
git switch main

# รวม feature เข้า main (ยืนอยู่บน main แล้วสั่ง merge)
git merge feature/login

# ลบ branch ที่ merge เสร็จแล้ว (เก็บกวาด)
git branch -d feature/login
git push origin --delete feature/login   # ลบบน remote ด้วย
💡

switch vs checkout: git checkout ตัวเก่าทำได้หลายอย่างจนสับสน (เปลี่ยน branch ก็ได้ กู้ไฟล์ก็ได้) Git รุ่นใหม่จึงแยกเป็น git switch (สลับ branch) กับ git restore (กู้ไฟล์) ให้ชัดขึ้น ใช้ตัวใหม่จะอ่านง่ายกว่า แต่เจอ checkout ในที่ทำงานก็ไม่ต้องตกใจ มันยังใช้ได้

06

ทำงานเป็นทีม & Pull Request

นี่คือสิ่งที่คุณจะทำจริงทุกวันในบริษัท — วงจรชีวิตของ PR

Pull Request (PR) คือ "คำขออนุญาตรวมโค้ด" คุณบอกทีมว่า "ฉันทำฟีเจอร์นี้เสร็จแล้วบน branch ของฉัน ขอรวมเข้า main นะ ช่วยรีวิวหน่อย" PR คือจุดที่เกิด Code Review และตรวจ CI/CD

🔁วงจรชีวิตของ Pull Request

ฝั่ง command line
git switch -c feature/JIRA-123-add-report
# ...แก้โค้ด เพิ่มฟีเจอร์...
git add .
git commit -m "feat: add monthly sales report export"
git push -u origin feature/JIRA-123-add-report
# จากนั้นไปเปิด PR บนหน้าเว็บ GitHub (มีปุ่มขึ้นให้กดอัตโนมัติ)

👀มารยาทใน Code Review (ทำให้ทีมรักคุณ)

👍ควรทำ

• PR เล็กๆ ทำทีละเรื่อง รีวิวง่าย
• เขียน title/description ให้ชัดว่าแก้อะไร ทำไม
• ตอบคอมเมนต์รีวิวทุกอัน
• ตั้งชื่อ branch สื่อความหมาย

👎ควรเลี่ยง

• PR ยักษ์ 3,000 บรรทัด ไม่มีใครอยากรีวิว
• commit รวมหลายเรื่องใน PR เดียว
• title แบบ "fix" / "update" เฉยๆ
• merge เองโดยไม่รอรีวิว (ถ้าทีมไม่อนุญาต)

🔃อัปเดต branch ให้ตามทันก่อน merge

ระหว่างที่คุณทำฟีเจอร์ คนอื่นอาจ merge เข้า main ไปแล้วหลายอัน ก่อนเปิด/merge PR ควรดึงของใหม่จาก main มารวมในแบรนช์ตัวเองก่อน:

terminal
# วิธีง่าย (merge main เข้ามา — ปลอดภัย เหมาะกับมือใหม่)
git switch feature/login
git merge main

# หรือวิธี rebase (ประวัติสะอาดกว่า แต่ระวังถ้า push ไปแล้ว)
git pull --rebase origin main
07

แก้ Merge Conflict (อย่ากลัว!)

มันคือเรื่องปกติที่เกิดขึ้นทุกวัน ไม่ใช่ความผิดพลาด แค่ Git ขอให้เราตัดสินใจ

Conflict เกิดเมื่อ สองคนแก้ไฟล์เดียวกันบรรทัดเดียวกัน Git ตัดสินใจแทนเราไม่ได้ จึงหยุดและใส่ "เครื่องหมาย" ไว้ให้เราเลือกว่าจะเอาเวอร์ชันไหน หน้าตาแบบนี้:

ไฟล์ที่เกิด conflict
<<<<<<< HEAD
    var greeting = "สวัสดีครับ";        // โค้ดฝั่งเรา (branch ปัจจุบัน)
=======
    var greeting = "Hello!";            // โค้ดที่กำลัง merge เข้ามา
>>>>>>> main

ขั้นตอนแก้ทีละสเต็ป

  1. เปิดไฟล์ที่ conflict (Git จะบอกว่าไฟล์ไหนใน git status)
  2. ลบเครื่องหมาย <<<<<<< , ======= , >>>>>>> ออก
  3. เก็บโค้ดที่ถูกต้องไว้ (อาจเอาฝั่งใดฝั่งหนึ่ง หรือผสมทั้งสอง)
  4. git add ไฟล์นั้น แล้ว git commit เพื่อจบการ merge
terminal
git status                 # ดูว่าไฟล์ไหน conflict
# ...แก้ไฟล์ในเอดิเตอร์ให้เรียบร้อย...
git add path/to/file.cs
git commit               # จบการ merge (ข้อความ default ก็พอ)

# ถ้าอยากยกเลิกการ merge ทั้งหมด กลับไปจุดเริ่ม
git merge --abort
git rebase --abort       # ถ้ากำลัง rebase อยู่
✅

เครื่องมือช่วย: VS Code มี "Merge Editor" ปุ่ม Accept Current / Accept Incoming / Accept Both ให้กดเลือกได้เลย ส่วน GitHub Desktop ก็มีหน้าจอ resolve conflict ที่ชี้ให้เปิดในเอดิเตอร์แล้วกลับมากด Continue — ไม่ต้องนับ <<< เองให้ปวดหัว

⚠️

ลดโอกาสเจอ conflict: commit บ่อยๆ ทีละเล็ก, pull งานทีมเข้ามาบ่อยๆ, และอย่าปล่อย branch ทิ้งไว้นานหลายสัปดาห์ ยิ่ง branch อยู่นาน ยิ่งห่างจาก main มาก conflict ยิ่งเยอะ

08

คำสั่งระดับโปรที่ใช้ในที่ทำงานจริง

เมื่อ add/commit/push พื้นฐานไม่พอ คำสั่งกลุ่มนี้คือของจริงที่แยกมือใหม่ออกจากมือโปร

📦git stash — พักงานไว้ก่อน

กำลังแก้ค้างอยู่ แต่ต้องสลับไปทำงานด่วนอื่น? stash เก็บงานที่ยังไม่ commit ไว้ในที่ปลอดภัยชั่วคราว ทำให้ working directory สะอาด:

terminal
git stash                # เก็บงานค้างไว้ (กลับสู่สภาพสะอาด)
git stash list           # ดูรายการที่เก็บไว้
git stash pop            # เอางานล่าสุดกลับมา + ลบออกจาก stash
git stash apply          # เอากลับมาแต่ยังเก็บไว้ใน stash

↩️reset vs revert — สองวิธี "ย้อน" ที่ต่างกันมาก

คำสั่งทำอะไรปลอดภัยกับงานทีมไหม
git revertสร้าง commit ใหม่ที่ "กลับ" การเปลี่ยนแปลงของ commit เก่า ประวัติยังอยู่ครบ✅ ปลอดภัย ใช้กับ branch ที่ push แล้วได้
git reset"ลบ" commit ออกจากประวัติ ย้อน HEAD กลับไป (เหมือนไม่เคยเกิด)⚠️ อันตรายถ้าทำกับ commit ที่ push ไปแล้ว
reset 3 ระดับ — จำความต่างให้ดี
# --soft : ลบ commit แต่เก็บการแก้ไว้ใน staging (พร้อม commit ใหม่)
git reset --soft HEAD~1

# --mixed (ค่า default): ลบ commit, เอาการแก้ออกจาก staging แต่ไฟล์ยังอยู่
git reset --mixed HEAD~1

# --hard : ลบ commit + ลบการแก้ทิ้งหมด!! (ระวังที่สุด ใช้เมื่อแน่ใจ)
git reset --hard HEAD~1

# วิธีปลอดภัยสำหรับงานที่ push แล้ว — ย้อนแบบมีหลักฐาน
git revert <commit-hash>

🍒cherry-pick — หยิบเฉพาะ commit ที่ต้องการ

อยากได้แค่ commit เดียวจาก branch อื่น (เช่นเอา hotfix ไปใส่อีก branch) โดยไม่เอาทั้ง branch:

terminal
git cherry-pick <commit-hash>

✏️amend — แก้ commit ล่าสุด

terminal
# ลืมใส่ไฟล์ / พิมพ์ข้อความผิด — แก้ commit ล่าสุดได้
git add forgotten-file.cs
git commit --amend -m "feat: add login (แก้ข้อความใหม่)"
🛑

กฎทอง: อย่าใช้ --amend, reset --hard, หรือ rebase กับ commit ที่ push ขึ้น branch ที่ทีมใช้ร่วมกันไปแล้ว เพราะมันเขียนประวัติใหม่ จะทำให้เพื่อนร่วมทีมพังตามไปด้วย ใช้กับงานที่ยังอยู่ในเครื่องเราเท่านั้น

🔀rebase — จัดประวัติให้เป็นเส้นตรงสวยงาม

Merge สร้าง "commit รวม" เพิ่มเข้ามา ส่วน rebase ย้ายฐานของ branch ไปต่อท้าย main ทำให้ประวัติเป็นเส้นตรง อ่านง่าย Interactive rebase ยังใช้รวม/แก้/เรียง commit ก่อนเปิด PR ได้:

terminal
# รวม/แก้ไข 3 commit ล่าสุดให้เรียบร้อยก่อนเปิด PR
git rebase -i HEAD~3
# ในหน้าจอที่เปิดมา เปลี่ยน "pick" เป็น:
#   squash = รวมเข้ากับ commit ด้านบน
#   reword = แก้ข้อความ commit
#   drop   = ลบ commit ทิ้ง
09

"แย่แล้ว!" — คู่มือกู้คืนตอนพลาด

ข่าวดี: Git แทบจะกู้คืนทุกอย่างได้ ถ้าเคย commit ไว้ ของไม่หายง่ายๆ

เผลอ commit ลง main แทนที่จะเป็น feature branch

terminal
# ยังอยู่บน main, ยังไม่ push
git switch -c feature/correct-branch   # สร้าง branch ใหม่จากจุดนี้ (พา commit ไปด้วย)
git switch main
git reset --hard HEAD~1               # เอา commit ออกจาก main

commit ล่าสุดข้อความผิด หรืออยากเอา commit ออกแต่เก็บโค้ดไว้

terminal
# แค่แก้ข้อความ commit ล่าสุด
git commit --amend -m "ข้อความที่ถูกต้อง"

# เอา commit ออก แต่เก็บการแก้ไฟล์ไว้ (กลับมาแก้ต่อ/commit ใหม่)
git reset --soft HEAD~1

แก้ไฟล์มั่วไปหมด อยากย้อนกลับเป็นเวอร์ชันที่ commit ไว้ล่าสุด

terminal
# คืนไฟล์เดียว
git restore path/to/file.cs

# คืนทุกไฟล์ที่ยังไม่ commit (ระวัง: การแก้จะหายหมด)
git restore .
⚠️

คำสั่งนี้ลบการแก้ที่ ยังไม่ commit ทิ้งถาวร เช็กให้ดีก่อนรัน — ของที่ยังไม่ commit กู้ยากกว่าของที่ commit แล้วมาก

เผลอลบ branch ที่มีงานสำคัญไป! ยังกู้ได้ไหม?

terminal — reflog คือพระเอก
# reflog เก็บประวัติทุกการเคลื่อนไหวของ HEAD ไว้ (แม้ commit ที่ "หาย")
git reflog
# หา hash ของ commit สุดท้ายบน branch ที่ลบ แล้วสร้าง branch กลับมา
git branch recovered-branch <commit-hash>
✅

git reflog คืออาวุธลับ มันจดทุกอย่างที่ HEAD เคยชี้ไป commit ที่คิดว่าหายไปแล้วมักจะยังอยู่ที่นี่หลายวัน

push commit ผิดขึ้น remote ไปแล้ว (แต่เป็น branch ของเราเอง)

terminal
# วิธีปลอดภัยที่สุด: revert (สร้าง commit ย้อน ไม่เขียนประวัติใหม่)
git revert <commit-hash>
git push

# ถ้าจำเป็นต้องเขียนประวัติใหม่บน branch ส่วนตัว ใช้ --force-with-lease
# (ปลอดภัยกว่า --force เพราะเช็กก่อนว่าไม่ทับงานคนอื่น)
git push --force-with-lease
🛑

อย่า force push ขึ้น branch ส่วนกลาง (main/develop) เด็ดขาด เว้นแต่ได้ตกลงกับทีมแล้ว มันลบงานคนอื่นได้

มาตรฐาน commit & branch ที่ทีมมืออาชีพใช้

เขียน commit ให้ดูเป็นมือโปรตั้งแต่ commit แรก — สร้างความประทับใจได้จริง

📝Conventional Commits — รูปแบบข้อความมาตรฐาน

รูปแบบ type: คำอธิบายสั้นๆ เป็นที่นิยมทั่วโลก ช่วยให้อ่านประวัติง่ายและทำ changelog/version อัตโนมัติได้:

Typeใช้เมื่อตัวอย่าง
featฟีเจอร์ใหม่feat: add export to Excel

refactorปรับโค้ดโดยไม่เปลี่ยนพฤติกรรมrefactor: extract pricing service

choreงานจิปาถะ (อัปเดต lib, config)chore: bump dependencies

✅

เคล็ดลับเขียน commit ดี: เขียนเป็นประโยคคำสั่ง (imperative) ภาษาอังกฤษ เช่น "add", "fix", "remove" ไม่ใช่ "added"/"fixed" — ลองอ่านว่า "ถ้า apply commit นี้ มันจะ ___" ให้ลงท้ายประโยคได้พอดี

🌿การตั้งชื่อ branch

รูปแบบที่นิยม

🗺️กลยุทธ์การ branch ของทีม

🌊GitHub Flow (นิยมที่สุดตอนนี้)

เรียบง่าย: มี main เป็นตัวจริง ทุกงานแตก feature branch สั้นๆ → เปิด PR → merge → deploy เหมาะกับทีมที่ deploy บ่อย (CI/CD) มือใหม่เริ่มจากตรงนี้ได้เลย

🏗️Git Flow (แบบดั้งเดิม)

ซับซ้อนกว่า: มีทั้ง main, develop, feature, release, hotfix เหมาะกับซอฟต์แวร์ที่ปล่อยเป็นรอบเวอร์ชัน (เช่น เดสก์ท็อปแอป) แต่หนักไปสำหรับเว็บที่ deploy ทุกวัน

💡

แต่ละบริษัทมีกติกาของตัวเอง วันแรกที่เข้างานให้ถามทีมว่าใช้ flow ไหน และ branch หลักชื่ออะไร (main/master/develop) อย่าเดาเอง เพราะกฎ branch ผิดอาจกระทบทั้งทีม

GitHub Desktop ฉบับใช้งานจริง

ทุกปุ่มในแอป = คำสั่ง Git ตัวไหน — เข้าใจแล้วใช้ได้ทันที

GitHub Desktop เหมาะมากสำหรับงานประจำวัน (commit, push, pull, สลับ branch, ดู diff) เพราะเห็นภาพชัดและพิมพ์ผิดยาก ส่วนงานซับซ้อน (interactive rebase, cherry-pick หลายตัว) ยังต้องพึ่ง command line อยู่บ้าง — หลายคนใช้ทั้งคู่

ในแอป GitHub Desktopเทียบเท่าคำสั่ง Git
File → Clone repositorygit clone
ช่อง "Changes" + ติ๊กถูกหน้าไฟล์คือ Working Directory + git add (ติ๊ก = staging)
ช่อง Summary + ปุ่ม "Commit to main"git commit -m "..."
ปุ่ม "Push origin"git push
ปุ่ม "Fetch/Pull origin"git fetch / git pull
Current Branch ▾ → New Branchgit switch -c
ปุ่ม "Create Pull Request"เปิดหน้าเว็บ GitHub เพื่อสร้าง PR

Branch menu → Stash changesgit stash
หน้าจอ resolve conflictช่วยจัดการ merge conflict แบบมี UI

🔄workflow ทั่วไปใน GitHub Desktop (ทีละขั้น)

  1. Clone โปรเจกต์จาก GitHub (File → Clone repository)
  2. กด Current Branch → New Branch ตั้งชื่อ feature ของคุณ
  3. แก้โค้ดในเอดิเตอร์ปกติ — ไฟล์ที่เปลี่ยนจะโผล่ในแท็บ Changes อัตโนมัติ
  4. ติ๊กไฟล์ที่จะ commit, เขียน Summary สั้นๆ, กด Commit to [branch]
  5. กด Publish branch / Push origin เพื่อส่งขึ้น GitHub
  6. กด Create Pull Request → ระบบเปิดหน้าเว็บให้กรอกรายละเอียด PR
✅

ข้อดีที่สุดของ GitHub Desktop: มันแสดง diff สีเขียว/แดง ให้เห็นชัดทุกบรรทัดก่อน commit ทำให้ตรวจทานง่ายและไม่เผลอ commit ไฟล์ที่ไม่ตั้งใจ — เป็นนิสัยที่ดีมากสำหรับมือใหม่

⚠️

UI และตำแหน่งเมนูของ GitHub Desktop รวมถึงหน้าเว็บ GitHub อาจเปลี่ยนตามเวอร์ชัน ขั้นตอนข้างบนเป็นภาพรวม flow ที่เสถียร แต่ชื่อปุ่ม/ตำแหน่งอาจต่างเล็กน้อย เช็กคู่มือทางการล่าสุดได้ที่ docs.github.com/desktop

เตรียมตัวสัมภาษณ์ & วันแรกของงาน

สิ่งที่บริษัทคาดหวัง + คำถามที่เจอบ่อย + กับดักที่ห้ามตกหลุม

🎤คำถามสัมภาษณ์ที่เจอบ่อย (ตอบให้ได้)

Q: Git กับ GitHub ต่างกันยังไง?

Git = เครื่องมือ version control ในเครื่อง, GitHub = บริการเก็บ repo บนคลาวด์ + ระบบทำงานร่วมกัน (PR, review, CI)

Q: merge กับ rebase ต่างกันยังไง?

merge รวม branch โดยสร้าง merge commit (ประวัติแตกกิ่ง), rebase ย้ายฐาน branch ไปต่อท้าย ทำให้ประวัติเป็นเส้นตรง แต่เขียนประวัติใหม่

Q: reset กับ revert ต่างกันยังไง?

revert สร้าง commit ใหม่เพื่อย้อน (ปลอดภัยกับงานที่ push แล้ว), reset ลบ commit ออกจากประวัติ (อันตรายกับ branch ส่วนกลาง)

Q: Staging Area คืออะไร?

พื้นที่กลางระหว่างไฟล์ที่แก้กับ commit ใช้เลือกว่าจะเอาไฟล์ไหนเข้า commit รอบนี้ จัดการด้วย git add

Q: จะแก้ merge conflict ยังไง?

เปิดไฟล์ที่ conflict, เลือกโค้ดที่ถูก, ลบเครื่องหมาย <<</===/>>>, แล้ว add + commit

Q: HEAD คืออะไร?

ตัวชี้ (pointer) ที่บอกว่าตอนนี้เราอยู่ที่ commit/branch ไหน — เหมือน "ตำแหน่งปัจจุบัน" ในประวัติ

🚩กับดักที่ทำให้ดู "มือใหม่" (เลี่ยงให้หมด)

  • commit bin/, obj/, node_modules/ หรือไฟล์ build ขึ้น repo (ลืมตั้ง .gitignore)
  • commit รหัสผ่าน/secret/connection string จริง — ผิดร้ายแรงด้านความปลอดภัย
  • ข้อความ commit แบบ "fix", "update", "asdf" — สื่อความหมายไม่ได้
  • git push --force ขึ้น branch ส่วนกลางจนลบงานเพื่อน
  • commit ก้อนใหญ่รวมหลายเรื่องในครั้งเดียว — รีวิวยาก ย้อนยาก
  • ทำงานบน main ตรงๆ โดยไม่แตก branch

หมายเหตุเรื่องความถูกต้อง: เนื้อหาคำสั่ง Git และแนวคิดหลักในคู่มือนี้เป็นความรู้มาตรฐานที่เสถียรและตรวจทานตามพฤติกรรมของ Git ที่ใช้กันทั่วไป (ตรวจสอบเทียบกับเอกสารทางการ git-scm.com ได้) ส่วนที่เกี่ยวกับ หน้าตา UI/ตำแหน่งปุ่มของ GitHub Desktop และหน้าเว็บ GitHub อาจเปลี่ยนแปลงตามเวอร์ชัน จึงควรยึดคู่มือทางการล่าสุดเป็นหลัก · กฎ branching/commit แต่ละบริษัทอาจต่างกัน ให้ยึดมาตรฐานของทีมที่คุณเข้าทำงานเป็นสำคัญ
